# Test-suite audit — 2026-07

Status: **Phase 1 (inventory and measurement)** — gate G1.
Scope: reduce total CI time while keeping exactly the coverage we need, with
a specific mandate to speed up the ASan+UBSan (Linux clang) job without
degrading its value as a memory-error and undefined-behavior detector.

All timings in this document are measured, not estimated: CI numbers come
from the GitHub Actions API over the last ~10 runs of `ci.yml` on `main`
(2026-07-22 → 2026-07-23); local numbers from a clean checkout of `main`
@ `1ca2ffd` on the development machine (macOS, arm64, 8 performance cores).

### 2.2 Where the sanitize job's time actually goes

Composition of the job (ccache-hot floor ≈ 13.5 min):

1. **Soak step: 9 m 02 s, fixed, every run (67 % of the hot-run floor).**
   `roadmaker_soak --minutes 9` with a run-id-derived seed. This is the
   in-CI random-op soak (hardening sprint §4.3) — distinct from the manual
   24-hour release-gate soak, which is out of scope for this audit.
2. **Build: 35 s (hot) → 14 m (every-file change).** ccache is configured
   and working; the variable cost is genuine instrumented compilation of
   changed files. One unified build already carries both ASan and UBSan.
3. **Test step: ~3 min, serial.** `ctest --test-dir build` runs 1859
   tests one after another with **no `-j` flag** — CI runners have 4
   cores, 3 idle. The same is true of every `testPresets` entry in
   `CMakePresets.json`: none sets `execution.jobs`.
4. Fixed setup (apt, caches, Qt): ~1 min.

### 2.3 Per-test profile under sanitizers (CI run 30030706976)

Serial ctest total: **171.96 s wall / 170.0 s summed** across 1859 tests.
The distribution is extreme:

| Slice | Time | Share |
|---|---:|---:|
| `SoakSmoke.FixedSeedRunsClean` (400 random ops) | 51.4 s | 30 % |
| `SoakSmoke.SameSeedSameNetwork` (3 × 150 ops determinism contract) | 7.1 s | 4 % |
| Remaining 1857 tests | ~111 s | 65 % |

Only **4 tests take ≥ 0.5 s**; the median test is ~0.05 s. The 111 s tail
is almost entirely **per-process startup overhead** (one process per test
case via `gtest_discover_tests`; each editor test boots an offscreen
QApplication), multiplied ~5× by ASan interception.

Top suites by aggregate time under sanitizers (excluding SoakSmoke):

| Suite | Tests | Total |
|---|---:|---:|
| PropertiesPanel | 67 | 7.0 s |
| ContextMenu | 28 | 4.5 s |
| Matrix/TJunctionQuality | 88 | 3.9 s |
| JunctionControl | 41 | 3.2 s |
| EditOperations | 69 | 2.7 s |
| LibraryDrop | 45 | 2.7 s |

Even the "slowest" ordinary suites cost single-digit seconds; their
per-test cost is startup overhead, not test logic.

### 2.4 Non-sanitized baseline (same commit, ubuntu build+test job)

Same 1867 entries: **39 s wall / 33.8 s summed** —

### 2.5 Local measurements (macOS arm64, `main` @ `1ca2ffd`)

1915 ctest entries locally (`1ca2ffd` includes the terrain tests added
after the profiled CI run). LSan is Linux-only, so local sanitizer runs

| Configuration | Serial (`ctest`) | Parallel (`ctest -j 8`) |
|---|---:|---:|
| dev (RelWithDebInfo, no sanitizers) | 164.0 s | 29.8 s (**5.5×**) |

**Parallel-safety finding:** `-j 8` runs fail exactly two tests, in both
normal and sanitized configurations, and both pass in isolation:
`WelcomeWidgetTest.RecentGridListsOnlyExistingFilesAndEmitsPath` and
`WelcomeWidgetTest.RecentProjectTilesShowNameAndSceneCountAndEmitDir` —
the WelcomeWidget suite shares QSettings state across concurrently
running test processes. Enabling `ctest -j` anywhere in CI **requires
isolating that state first** (per-test `QSettings` scope, or a
`RESOURCE_LOCK`/serial-group property as a fallback). This is the
already-known WelcomeWidget parallel flake, now scoped precisely; it is
a blocker-class prerequisite for the main optimization lever.

## 3. Diagnosis — what actually makes the sanitize job slow

Ranked by impact on a typical PR run (~17–21 min job):

1. **The fixed 9-minute soak step** — the single largest constant (67 %
   of the hot-run floor). Pure execution; any reduction is a policy
   decision about how long the random-op soak runs per PR versus on
   merge to `main`, not an engineering trick.
2. **Instrumented build of changed files** (0.5–14 min, median ~4 min on
   C++ PRs) — ccache already absorbs unchanged files; the remaining cost
   is irreducible per-file compile time under `-fsanitize`.
3. **Serial ctest execution** (~3 min → parallelizable; the runner has 4
   cores). The per-test process model multiplies Qt startup 868× under
   ASan interception. Blocker: WelcomeWidget QSettings sharing (§2.5).
4. **Test *content* is a rounding error.** Removing any ordinary test or
   even whole suites cannot move this job meaningfully: outside SoakSmoke,
   the most expensive suite costs 7 s of 172 s. Duplicate/redundant test
   removal must therefore be justified by *maintenance cost and suite
   clarity*, not by sanitize-job wall time — the wall-time levers are
   (1)–(3).

Corollary for Phase 2: the rule "no test excluded from the sanitizer job
unless purely UI/rendering with no memory logic" costs almost nothing to
honor — broad coverage under ASan is cheap once execution is parallel;
the expensive items are the soak policy and the instrumented build.

## 6. Phase-2 inputs recorded during inventory

Observations gathered while classifying; **no action taken** — these are
inputs to the Phase 2 proposal, each to be argued there:

1. **Naming-collision clusters are real but behavioral overlap is low.**
   Round-trip/persistence family, the three "surface" features, tool-vs-
   kernel pairs, the signal family, and mesh_parity were each skimmed:
   suites are layered by concern (query / command / persistence / mesh /
   tool / binding) and no two files assert the same thing on the same
   code path. The genuine duplication found is (a) the recurring
   "empty/pre-feature network re-exports byte-identically" assertion
   repeated across several persistence suites, and (b) shared fixture
   setup, not shared assertions.
2. **SoakSmoke inside the sanitize job is double coverage by design**:
   `FixedSeedRunsClean` (51 s under ASan) runs in the same job that then
   soaks the identical driver for 9 minutes with a varied seed. On the
   three build+test jobs, SoakSmoke is the *only* soak-driver exercise
   and must stay. `SameSeedSameNetwork` is the determinism contract —
   unique behavior, not covered by the soak step.
3. **Stale retired-program labels**: three tests cite the retired GS-1
   golden-scene name for live fixtures (test_junction_crosswalks,
   test_camera's golden pose, python test_road_marks). Behavior is
   current; only the label is dated. Rename-in-place candidates.
4. **Vendor-name ban: clean.** Zero matches for competitor product names
   across all three test trees.
5. **No tests exercise archived behavior.** Every feature the archive
   retired (milestone releases, golden scenes as a program) has no
   surviving behavioral test; M3a/GS-1 appearances are comment
   provenance only.
6. **8 disabled tests** are the `*.WriteCorpusSeed` fixtures regenerators
   — intentional utilities, correctly disabled, not dead weight.

# Proposal (Phase 2) — gate G2

Target metric up front:

| Scenario | Today | Proposed | Mechanism |
|---|---:|---:|---|
| Sanitize job, ccache-hot PR | ~13.4 min | **~6 min** | 3-min PR soak + parallel ctest |
| Sanitize job, typical C++ PR | 16.5–21 min | **~9–13 min** | same + unchanged build cost |
| Sanitize job, push to `main` | ~13.4–27 min | ~12 min hot | full 9-min soak retained |
| build+test jobs' Test steps | 39 s–3 min | ÷3–4 | parallel ctest |

After this, the sanitize job leaves the PR critical path — pipeline
wall-clock becomes bounded by `build+test windows-latest` (~17 min),
which is untouched by this audit.

## C. ASan+UBSan job optimization plan

Ordered by the Phase-1 diagnosis (§3). Every change preserves the job's
detection value: same sanitizers, same flags, same test population under
instrumentation on every PR (one documented exception, C3).

**C1 — Parallelize ctest (all jobs).** Add `-j 4` to the four CI ctest
invocations (three `ctest --preset ci-*` + the sanitize
`ctest --test-dir build`). Runners have 4 cores; ctest currently uses 1.
Measured at 8 local cores: 5.5× (normal) / 4.2× (ASan); at 4 cores
expect ~3×. Sanitize Test step ~3 min → **~1 min**. No preset changes —
developers keep opting in with `ctest -j N`; the contributing docs gain
a line.
*Prerequisite:* fix the WelcomeWidget QSettings sharing (the only
parallel failure in 4 local `-j 8` runs across both configurations —
scoped in list B). Phase 3 verification: 5 consecutive `-j` runs of both
configurations must be green before the CI flag lands, and any further
parallel-unsafe test found gets fixed or `RESOURCE_LOCK`-serialized in
the same commit.

**C2 — Split the in-CI soak: 3 minutes on PRs, 9 on `main`.** The
9-minute varied-seed soak is 67 % of the hot job and runs identically on
every PR push. Proposal: `--minutes 3` when `github.event_name ==
'pull_request'`, keep `--minutes 9` on push to `main` (trigger already
exists in `ci.yml` — no new tier infrastructure, so the "tier defined
but never executed" failure mode cannot occur). Rationale: PR soaks are
cumulative across pushes (seed varies per run), every merge re-soaks at
full length on `main`, and the manual 24-hour release-gate soak is
untouched. This is the only change in the plan that trades sanitizer
*exposure time* for wall time, and it is a policy call — flagged for
explicit maintainer sign-off at this gate.
*Alternative if 9-min PR soaks must stay:* run the soak concurrently
with ctest in one step (`soak & ctest -j3; wait`) — keeps full exposure,
saves only the ~1-min ctest overlap (hot job ~11 min instead of ~6).

**C3 — Skip `SoakSmoke.FixedSeedRunsClean` in the sanitize job only**
(`-E 'SoakSmoke\.FixedSeedRunsClean'`), saving its 51 s (30 % of today's
Test step). Justification: in that job, and only in that job, the very
next step runs the *same driver* under the *same sanitizers* for 3–9
minutes — the fixed-seed 400-op run is a strict subset of coverage
there. It continues to run under sanitizers' absence on the three
build+test jobs (its original purpose: every CI run exercises the
driver) and in every local run. `SoakSmoke.SameSeedSameNetwork` (the
determinism contract) is unique behavior and stays everywhere. This is
the plan's single test-selection exception, documented here per the
audit's selection-not-exclusion rule.

**C4 — Build: verified, no action.** ccache is present and effective
(35 s hot builds); ASan and UBSan already share one build; configure is
seconds. The remaining variable cost is genuine instrumented compilation
of changed files. Considered and rejected: batching test cases into
fewer processes (per-suite `add_test`) — it would cut the ~110 s
per-process overhead but degrades ctest granularity/reporting and is
superseded by C1's parallelism; a separate soak job sharing the build
via artifacts — instrumented-binary upload costs more than it saves.

## A. Candidates for removal or merging

**Result: none.** A dedicated adversarial pass examined every lead from
the Phase-1 overlap characterization against the strict criteria (same
code path + same effective assertions; strict subset of an always-run
test; behavior that no longer exists). Findings, so the negative result
is auditable:

| Lead examined | Verdict | Evidence |
|---|---|---|
| "Absent feature re-exports byte-identically" across 9 persistence suites | keep all | each asserts a *different* feature's absence-encoding against its own writer path (`corners.empty()`, `stoplines.empty()`, `maneuvers.empty()`, `phases.empty()`, …); the shared shape is the acceptance contract, the assertions are disjoint |
| test_smoke.cpp (7 cases) | keep all | seven unique third-party link/behavior checks (Eigen, Clothoids, Clipper2, CDT, Manifold, pugixml, version string); none creates a network, none subsumed; the Manifold case is documented as a deliberately-kept call site |
| camera / framing / projection / nav_controller | keep all | four SUT layers (matrix math, world→screen mapping, selection bounds, input chords); the apparent ortho overlap asserts matrix vs mapping — different SUT |
| test_panels vs dedicated panel files | keep all | different widget classes; the nearest pair drives different commands (`SetLaneWidth` vs `SetLaneWidthProfile`) and a test exists precisely to distinguish them |
| picking / select_tool / highlight | keep all | hit-result vs undo-side-effect vs highlight-set assertions on three layers |
| malformed-input parameter lists | keep all | every value hits a distinct reject branch except one pair: `dur=-4` duplicates the `dur=0` boundary sub-condition (and `dur=inf` collapses with `dur=3601` onto the max-guard). `dur=0`/`dur=inf` are legitimate boundary/IEEE edges; per "when in doubt, keep" all stay |
| files superseded by refactors | none exist | the two historical cases (node-drag tool, diagnostic locator) were already deleted with their refactors |
| pytest duplicating core gtests | keep all | every pytest case exercises binding surface (exception translation, read-only views, `__repr__`, Python save/load API) |

This is consistent with the timing data: outside SoakSmoke no suite
costs more than 7 s even under sanitizers, so removals were never a
wall-time lever — and the suite turns out to be genuinely layered, not
duplicated. **Zero tests are proposed for removal.**

One cleanup rides along (relabeling, not removal): three tests cite the
retired GS-1 golden-scene program by name for live fixtures
(`test_junction_crosswalks.cpp`, `test_camera.cpp`'s golden pose,
`python/tests/test_road_marks.py`). The comments/docstrings get renamed
to describe the fixture ("4-arm urban intersection", "diagonal
showcase camera"); assertions untouched.
