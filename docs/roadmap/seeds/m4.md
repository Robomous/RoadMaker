# M4 seed — scenario mode

*Scope sketch for M4. A seed, not a design: full design docs are written
when M4's planning task runs.*

- **Theme:** RoadMaker authors scenarios, not just maps.
- **Golden scene:** [GS-3 "Ambulance run"](../golden_scenes/gs3_ambulance_run.md)
- **Release target:** v0.6.0
- **Gap coverage:** [gap 2 — asset library browser](../gap_analysis.md#gap-2--asset-library-browser),
  [gap 3 — scenario editing mode](../gap_analysis.md#gap-3--scenario-editing-mode),
  [gap 5 — application modes](../gap_analysis.md#gap-5--application-modes)

## Scope sketch

### Kernel

- **OpenSCENARIO XML 1.4.0 model** in `core/xosc/`: entities (vehicles,
  pedestrians), storyboard basics, routes/trajectories referencing the road
  network (road/lane ids + s/t), initial actions (speed, lane offset).
  Reader/writer with the same diagnostics-first posture and rule-id
  citations as the xodr path; normative chapters read before implementing
  ([references](../../domain/references.md)).
- Route anchoring model: lane-anchored polylines with per-segment lateral
  offsets, resilient to map edits (anchors by id + s, not world coords).
- Python bindings + examples for the scenario model (same-PR rule).

### Editor

- **App mode switch (Map ↔ Scenario)** — decides gap 5. Options to present
  in design (deliberately NOT decided in this seed):
  - *Toolbar-level mode toggle:* one window, one layout; panels/tools swap
    in place. Cheaper, keeps one dock-layout state.
  - *Separate window layouts per mode:* each mode owns a saved dock layout
    (Qt `saveState`/`restoreState` per mode); heavier, closer to
    commercial-editor feel.
  Either way: mode = which tools, panels, and selection types are active;
  `Document` gains a scenario sub-document; `SelectionModel` learns
  scenario entity kinds.
- **Scenario tools:** actor placement (drag from library), route drawing
  with lane anchoring and offset handles, actor attributes panel
  (QAbstractItemModel-backed, tester in same commit).
- **Asset Library Browser** — mode-agnostic infrastructure serving props/
  materials in map mode and vehicles in scenario mode:
  - *Manifest:* extend `assets/manifest.json` (or a sibling
    `library.json`) with runtime metadata — category path, display name,
    preview image ref, placement semantics (prop vs vehicle vs material).
  - *Thumbnails:* pre-rendered at asset-import time by a script (offscreen
    render or packaged previews), cached under the asset tree; no live
    GL thumbnailing inside panels.
  - *Panel:* `QAbstractItemModel`-based tree/grid with search filter
    (`QSortFilterProxyModel`), drag-to-place via Qt drag/drop onto the
    viewport; follows every editor architecture rule
    ([editor](../../architecture/editor.md)).

### Assets

- CC0 vehicle set incl. an ambulance-class model
  ([candidates](../asset_candidates.md)); pedestrian models optional.

## Dependencies on prior milestones

- M3a's `<objects>` model (actors placed relative to a visually complete
  map; props/vehicles share placement machinery).
- M2 command layer — every scenario edit is an undoable command.
- GS-1 as the GS-3 map underlay.

## Top 3 risks

1. **Two-document coupling** (map + scenario): map edits invalidating
   scenario anchors is the classic failure; mitigation: id+s anchoring,
   validation pass that reports dangling anchors as diagnostics, tests that
   edit the map under a scenario.
2. **Mode switch retrofit cost** — bolting modes onto a single-mode
   MainWindow could sprawl; mitigation: decide the mode architecture at
   design time (the two options above), before any scenario tool lands.
3. **OpenSCENARIO scope creep** — the standard is huge; mitigation: model
   exactly what GS-3 requires (entities, one route, offsets, speed, minimal
   storyboard), design the schema for extension in M5.

## Open questions for the maintainer

1. Mode switch: toolbar-level toggle or per-mode window layouts? (Options
   above; decision belongs to the M4 design task.)
2. Library manifest: extend `assets/manifest.json` or introduce a separate
   runtime `library.json` (build-time provenance vs runtime catalog have
   different lifecycles)?
3. Do pedestrians ship in M4, or is GS-3's single vehicle actor the whole
   entity scope?
4. Scenario file pairing: is `.xosc` saved beside `.xodr` with a relative
   reference (OpenSCENARIO's RoadNetwork element), or does RoadMaker gain a
   project-file notion? (`.xodr`-as-project was an M2 decision; scenarios
   strain it.)
