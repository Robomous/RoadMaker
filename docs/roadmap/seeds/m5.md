# M5 seed — scenario logic

*Scope sketch for M5. A seed, not a design: the node editor's actual node
graph, canvas interactions, and schema mapping are deliberately NOT designed
here — that is M5's own planning task.*

- **Theme:** scenario *logic* becomes visually authorable.
- **Golden scene:** [GS-3 extended with a logic graph](../golden_scenes/gs3_ambulance_run.md#m5-extension--logic-graph)
- **Release target:** v0.10.0
- **Gap coverage:** [gap 4 — logic editor](../gap_analysis.md#gap-4--logic-editor)

## Scope sketch

### Kernel

- Complete the OpenSCENARIO storyboard model: stories, acts, maneuver
  groups, maneuvers, events, actions, and the condition system (trigger
  conditions, condition groups, edges/rising-falling semantics) — to the
  depth the logic editor exposes, not the whole standard.
- Validation with rule-id citations; round-trip stability for
  editor-authored storyboards.

### Editor

- **Node-based logic editor:** a canvas where stories/maneuvers/conditions
  are nodes with typed ports, transitions/triggers are edges, and the graph
  round-trips to the `.xosc` storyboard. Editing follows the same command/
  undo discipline as everything else.
- Graph ↔ document sync: selection in the graph highlights the affected
  actors/routes in the viewport (through `SelectionModel`).
- Validation feedback rendered on the graph (bad condition, unreachable
  state) from kernel diagnostics.

### Simulation preview hooks

- **esmini interop:** export-and-run against a headless esmini as a preview/
  smoke path (esmini is MPL-2.0 — passes the
  [dependency policy](../../standards/dependencies.md) if integrated;
  running it as an external process avoids linking entirely — preferred).
- CI: headless esmini run of the GS-3 scenario as an artifact-producing
  smoke test (screenshot/log artifact), building on the ecosystem
  round-trip idea from earlier plans.

## The node-editor technology question (open, decision in M5 design)

Candidates, all subject to the dependency policy:

- **Custom on QGraphicsScene/QGraphicsView** — zero new dependencies, full
  control, most implementation effort; Qt's canvas is built for exactly
  this class of UI.
- **Third-party Qt node libraries** (e.g. QtNodes and similar) — faster
  start, but each candidate must pass the
  [license policy](../../standards/dependencies.md) (several popular node
  libs are LGPL/GPL — LGPL would need the same scrutiny as any non-Qt LGPL
  candidate, which the policy says requires explicit maintainer approval)
  and the code-quality bar for a long-lived sub-product.
- **Web-canvas embed** is out — no WebEngine (weight, LGPL surface, breaks
  the sober-native posture).

The seed's stance: evaluate custom-QGraphicsScene as the default, treat any
library as the exception that must prove itself.

## Dependencies on prior milestones

- M4's OpenSCENARIO model, mode switch, and scenario tools — the logic
  editor is a third view on the M4 scenario document.
- M4's actor/route authoring: graphs reference entities that must already
  exist.

## Top 3 risks

1. **Sub-product scale** — a node editor is its own application-sized UI;
   mitigation: scope the node vocabulary to GS-3's story depth, ship
   read-only graph *rendering* before graph *editing* if needed.
2. **Semantic gap between graph and standard** — OpenSCENARIO's storyboard
   is a nested XML tree with condition wiring, not a free graph; a naive
   canvas lets users draw invalid structures; mitigation: the graph is a
   *projection* of the storyboard schema (typed ports, constrained edges),
   never a freeform diagram.
3. **esmini behavioral drift** — our validation passing but esmini
   rejecting (or vice versa) confuses users; mitigation: esmini smoke in CI
   from day one, divergences filed as explicit compatibility notes.
