# 03 — City props

*Buildings and streetlights, so a scene has a city around the road.*

## 1. The decision the seed made, and the one it did not

The seed's approved default is **"buildings = Library props, not procgen"**
(overview decision 3). That decision is about **product scope**: RoadMaker is not
getting a procedural building generator — no footprint extrusion tool, no
facade parameters, no city-block synthesis. Buildings are assets you drag in.
That stands, unchanged.

It does **not** settle where the meshes come from, and the repo's actual
precedent cuts against the obvious reading.

### What M3a actually shipped

Every prop in the editor today — the four trees, the shrub, the traffic light,
the sign — is **procedurally authored original work**, generated by
`scripts/gen_prop_meshes.py` into `prop_meshes.gen.cpp`, licensed **MIT** as
project assets. Not one is a Kenney or Quaternius download. The kits in
[asset_candidates.md](../../roadmap/archive/2026-07-pre-reset/asset_candidates.md) were scouted; the
implementation went its own way and the ledger shows it.

And `asset_candidates.md` carries a rule that now binds us:

> "Complementary style — **pick ONE primary style per scene** to avoid a
> mismatched look."

So dropping a Kenney City Kit next to those procedural trees is the exact
mismatch that rule exists to prevent — a stylised asset-store building beside a
hand-rolled low-poly tree reads as two products in one viewport.

### Decision: verify Kenney first, procedural fallback, style is the tiebreak

1. **Fetch and evaluate Kenney City Kit / City Kit (Suburban) + streetlights**
   at implementation time, per the asset standard (licence verified on the
   source page at fetch time — a doc mapping is "a candidate, not a fact, until
   the manifest row records the verification").
2. **Render them next to the existing props before committing.** If the styles
   sit together, use them: they are CC0, they are the seed's preference, and
   they cost nothing to maintain.
3. **If they clash, extend `gen_prop_meshes.py` instead.** Boxy low-poly
   buildings and a streetlight are well within what that script already does
   (frustum, cylinder, cone, icosahedron primitives are all there). This is
   *still* "Library props, not procgen" in the sense decision 3 meant — the
   generation happens at asset-build time on the developer's machine and the
   output is a fixed mesh in the library, not a feature the user drives.
4. **Record which path was taken** as an As-built note here, with the
   comparison render in the PR.

**Flagged for the maintainer** on the epic alongside the persistence split: if
the intent behind "Library props (CC0)" was specifically *"download from a CC0
kit, do not hand-author"*, say so and step 3 goes away. The evidence suggests
otherwise, but this is a taste call about how the product looks, which is the
maintainer's.

## 2. The kit

| Prop | Purpose | Source (candidate) |
|---|---|---|
| Building × ~4 (low/mid-rise, varied footprint) | GS-2's district; street walls behind sidewalks | Kenney City Kit, else procedural |
| Streetlight | GS-1-class urban scenes; the seed names it explicitly | Kenney, else procedural |
| Bench, bollard, litter bin *(if cheap)* | Street furniture density | Same |

Deliberately small. GS-4 (this milestone's golden scene) is **rural** and needs
none of these — city props are here because **GS-2 is unapprovable without
them** and this milestone is GS-2's prerequisite. Sizing the kit to "what GS-2
needs to be approvable" rather than "a city kit" is what keeps WS-4 from
becoming its own milestone.

### GS-2 coverage check

[GS-2 "Imported district"](../../roadmap/archive/2026-07-pre-reset/golden_scenes/gs2_imported_district.md)
is a ~500 m × 500 m OSM extract. Its buildings come from **OSM footprints**, not
from hand placement — which means WS-4's job is to supply the *meshes and the
library category*, while M3b supplies the *placement from footprint data*.

**This is a seam worth stating plainly:** if GS-2 renders buildings by extruding
OSM footprints, it needs a footprint-extrusion path, which is procedural
building geometry — the thing decision 3 excludes. Either GS-2 places kit props
at footprint centroids (loses the district's shape) or M3b gets a footprint
extruder (contradicts decision 3, and belongs in M3b's planning, not here).
**Not resolved here; raised on the epic and in M3b's seed** so M3b's planning
pass owns it rather than discovering it. WS-4 delivers the props either way.

## 3. Import path and layout

Reuses the M3a pipeline unchanged — no new machinery:

- `assets/manifest.json` gains rows (url / sha256 / licence / licence_url /
  author / destination / retrieved / postprocess); `scripts/fetch_assets.py`
  fetches; `scripts/check_asset_licenses.py` gates in CI (every file under
  `assets/` needs a ledger row, every licence must be in the allowed set);
  `ASSETS_LICENSES.md` gets a row per file.
- Procedural props instead go through `scripts/gen_prop_meshes.py` →
  `prop_meshes.gen.cpp`, ledgered as "original work (this repo)", MIT.
- `assets/library/manifest.json` gains a **Buildings** category (and adds
  streetlights to the existing **Props**). Item kind is the existing
  `Kind::Tree` mesh-instance path renamed/extended — a building is a mesh
  instance at a pose, exactly like a tree, so `add_object` and the instanced
  render path already handle it.
- Placement is the existing OpenDRIVE `<object>` mechanism. No new kernel model.

**Repo weight:** props are meshes, not textures, so the ≤512 KB/map budget does
not apply — but the same restraint does. Low-poly, no per-building textures in
v1 (they take a material from `01`'s library like any other surface).

## 4. Sign-text rendering — STRETCH

**Not committed scope** (overview decision 5, resolving the seed's open question
3). Specified here only so that if it is picked up, it is picked up correctly —
and so that "stretch" means something more than "maybe later".

**Mini-spec:**

- **What**: render text onto sign faces (a speed limit's number, a street name)
  by rasterising glyphs to a texture at load, then binding it as the sign's
  albedo.
- **Why it is stretch**: it needs a font, and a font needs a licence
  (SIL OFL is not on the allowed list — CC0/MIT/ISC/Apache-2.0 only — so this is
  a licence question before it is a rendering question). Qt can rasterise with a
  system font, but a system font is not redistributable and CI has no guarantee
  of one.
- **Route if taken**: rasterise with `QPainter` into a `QImage` → existing
  `TextureData` upload path → a `Material` with that albedo. No new renderer
  concepts; the whole cost is asset/licence, not code.
- **Gantry signs**: out of v1 entirely. A gantry is a structure (WS-3's
  vocabulary) carrying a sign (WS-4's), and it needs sign-text to be worth
  anything. If sign-text lands, gantries are a follow-up issue, not a slice.

**Do not start this until WS-1…WS-4 are done.** It is the first thing cut.

## 5. Tests

- Manifest parses the new category; an unknown item kind still degrades to
  `Unknown` (the forward-compat guarantee).
- A building places from the Library as an `<object>`, round-trips, and renders
  as an instance.
- `check_asset_licenses.py` passes — every new file has a ledger row with a
  verified licence.
- Screenshot CI: a props shot with buildings and streetlights, reviewed for the
  style-consistency call in §1.
